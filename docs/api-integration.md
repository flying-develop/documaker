# API Integration Documentation

## 1. Общий обзор сетевого слоя

Сетевой слой фронтенд-приложения `documaker-frontend` построен на основе **Redux Toolkit Async Thunks**, обеспечивающих типобезопасную, предсказуемую и легко тестируемую обработку асинхронных HTTP-запросов. Все API-интеграции инкапсулированы в сущностях (`Auth`, `Profile`), соответствующих доменным областям приложения.

Ключевые принципы:
- **Централизованная маршрутизация**: URL-адреса формируются через утилиту `getUrl()` с явным указанием сервиса и эндпоинта — это обеспечивает единообразие и упрощает миграцию/перенастройку базовых URL.
- **Изолированные бизнес-логики**: каждый `createAsyncThunk` отвечает за одну чётко определённую операцию (логин, регистрация, получение профиля и т.д.).
- **Типизированные параметры**: входные данные передаются как `Record<string, string | number>`, что позволяет гибко поддерживать как query-параметры, так и тело запроса (в зависимости от метода).
- **Единая обработка ошибок**: все исключения перехватываются и нормализуются через `rejectWithValue()`, обеспечивая согласованный формат ошибок в Redux-состоянии.

## 2. Используемый HTTP-клиент и базовая конфигурация

### HTTP-клиент
Используется **Axios** (`&/shared/api` — экспортирует настроенный экземпляр `api`).

### Базовая конфигурация
- **Базовые URL** для каждого сервиса определяются в `Services` (например, `Services.AUTH = 'https://api.documaker.dev/auth'`).
- **Эндпоинты** хранятся в объекте `Endpoints` как строковые константы (например, `Endpoints.LOGIN = '/login'`).
- **Формирование URL**:  
  - Для простых путей используется `getUrl({ service, endpoint })`.  
  - Для динамических путей (с параметрами в пути или query) применяется `combineUrl({ endpoint, params })`, который:
    - Добавляет query-параметры в URL (если `params` содержит ключи, не относящиеся к path-переменным),
    - Поддерживает шаблонизированные пути (например, `/managers/:id`) — *но в текущих файлах таких случаев не обнаружено*.
- **Автоматическая установка заголовков**:  
  - `Authorization: Bearer <token>` добавляется автоматически при наличии access token (реализовано в `&/shared/api`, но не видно в представленных файлах — подразумевается глобальный interceptor).
  - `Content-Type: application/json` установлен по умолчанию.

## 3. Основные сервисы и эндпоинты

### Сервисы
| Сервис | Назначение | Пример базового URL |
|--------|------------|---------------------|
| `Services.AUTH` | Аутентификация и управление учётными записями | `https://api.documaker.dev/auth` |
| `Services.PROFILE` | Управление профилем пользователя и токенами | `https://api.documaker.dev/profile` |
| `Services.PETITIONS` | Работа с делами (кейсами), заявками и документами | `https://api.documaker.dev/petitions` |

### Эндпоинты и их использование

| Сущность | Thunk | HTTP-метод | Путь (относительно сервиса) | Назначение | Примечания |
|----------|--------|-------------|------------------------------|------------|------------|
| `Auth` | `checkLogin` | `POST` | `/login` | Проверка существования пользователя по email/телефону | Используется для pre-login flow (например, выбор формы входа/регистрации) |
| `Auth` | `login` | `POST` | `/login/password` | Аутентификация по email/паролю | Требует `email`, `password` |
| `Auth` | `register` | `POST` | `/register` | Регистрация нового пользователя | Поля: `name`, `phone`, `email`, `password`, `password_confirmation` |
| `Auth` | `forgot` | `POST` | `/forgot` | Запрос восстановления пароля | Ожидает `email` |
| `Auth` | `reset` | `POST` | `/reset` | Сброс пароля по токену | Требует `email`, `token`, `password`, `password_confirmation` |
| `Profile` | `fetchProfile` | `GET` | `/` | Получение данных текущего пользователя | Без параметров или с опциональными query-параметрами |
| `Profile` | `updateProfileInfo` | `PATCH` | `/update` | Обновление профиля | Отправка тела запроса (поля для обновления) |
| `Profile` | `refreshAccessToken` | `POST` | `/refresh` | Обновление access token по refresh token | Тело запроса не требуется — токен передаётся в заголовке `Authorization: Bearer <refresh_token>` |
| `Profile` | `fetchManagers` | `GET` | `/managers` | Получение списка менеджеров | Поддерживает query-параметры (например, `page`, `limit`, `search`) |
| `Profile` | `fetchCases` | `GET` | `/cases` | Получение списка дел пользователя | Поддерживает query-параметры (например, `status`, `sort`, `page`) |

> ⚠️ **Примечание**: Путь `/` в `fetchProfile` и `/update` в `updateProfileInfo` — это конкретные значения из `Endpoints.PROFILE_UPDATE` и пустая строка (`''`) для основного профиля, разрешённые `combineUrl`.

## 4. Аутентификация, токены и refresh flow

### Модель токенов
Приложение использует **JWT-токены** с разделением ролей:
- `access_token`: короткоживущий токен для авторизации запросов к защищённым ресурсам (обычно 15–60 мин).
- `refresh_token`: долгоживущий токен для безопасного получения нового `access_token` без повторного ввода логина/пароля.

### Жизненный цикл аутентификации
1. **Логин** → `POST /auth/login/password`:  
   В ответе ожидается объект с `access_token`, `refresh_token`, `user` и, возможно, `expires_in`.
2. **Хранение токенов**:  
   Сохраняются в `localStorage` или `httpOnly` cookies (реализация вне scope файлов, но предполагается наличие secure-механизма).
3. **Автоматическая авторизация запросов**:  
   Axios interceptor добавляет `Authorization: Bearer <access_token>` ко всем запросам к `Services.PROFILE`, `Services.PETITIONS` и другим защищённым сервисам.
4. **Обработка истечения срока действия access token**:  
   При получении `401 Unauthorized` вызывается `refreshAccessToken` → `POST /profile/refresh`.  
   - Если успешен — новый `access_token` сохраняется и текущий запрос повторяется.  
   - Если `refresh` провалился (`401` или `403`) — пользователь перенаправляется на страницу входа.
5. **Ручной refresh**:  
   `refreshAccessToken` может быть вызван явно (например, при старте приложения или в фоновом режиме), чтобы продлить сессию.

### Безопасность
- `refresh_token` **никогда не отправляется в теле запроса** при обычных операциях — только в `refresh`-запросе и только в защищённом контексте (HTTPS, `httpOnly` cookie предпочтительно).
- Все чувствительные операции (`login`, `register`, `reset`) требуют CSRF-защиты (подразумевается на бэкенде; фронтенд должен обрабатывать `X-CSRF-TOKEN` при необходимости — не отражено в коде, но требует документирования для бэкенда).

## 5. Форматы данных и обработка ошибок

### Форматы запросов и ответов
- **Content-Type**: `application/json` (по умолчанию).
- **Запросы**:  
  - `POST`/`PATCH`: тело — JSON-объект (`{ "key": "value" }`).  
  - `GET`: параметры передаются как query-строка (через `combineUrl({ params })`).
- **Ответы**:  
  - Успешные: `200 OK` или `201 Created` с JSON-телом, содержащим данные (`data`-поле в ответе Axios).  
  - Ошибки: стандартные HTTP-статусы (`400`, `401`, `403`, `422`, `500`), тело — объект ошибки вида:  
    ```json
    { "message": "Validation failed", "errors": { "email": ["The email field is required."] } }
    ```

### Обработка ошибок
- Все thunks используют единый паттерн `try/catch` + `rejectWithValue(err)`.
- `err` передаётся «как есть» — это может быть:
  - `AxiosError` (с полями `response.status`, `response.data`, `request`, `message`),
  - Сетевая ошибка (`err.code === 'ERR_NETWORK'`),
  - Ошибка таймаута.
- В `rejected`-действии Redux рекомендуется проверять:
  - `error.response?.status` для HTTP-статусов,
  - `error.response?.data` для структурированных ошибок бэкенда,
  - `error.message` для клиентских/сетевых ошибок.

### Валидация и сериализация
- На стороне фронтенда **нет строгой валидации входных параметров** — полагается на договорённости с бэкендом.
- Рекомендуется дополнять thunks схемами (например, через `zod` или `yup`) для раннего выявления ошибок ввода.
- Поля ввода (`confirmPassword`) маппятся в `password_confirmation` вручную (см. `register`, `reset`) — это требует синхронизации с бэкендом.
